# LLM prompt — behavior analyzer (sandbox JavaScript)

Use this template with any LLM to generate analyzer code for DSTB **sandbox** mode. Paste the resulting JavaScript into the dashboard Analyzer Editor, run **Test Run** against a raw cycle, then save.

## Contract

1. The host wraps your code and calls `analyze(input)` after attaching helpers.
2. **You must define:** `function analyze(input) { ... }`
3. **Return value:** `{ label: string, details: object }`
   - `label`: short string shown in behavior results tables (e.g. `"ATT_BGN_EARLY"`, `"NEUTRAL"`).
   - `details`: any JSON-serializable object for debugging (counts, sub-labels, etc.).

## `input` shape

```ts
// input.candles — arrays of { t, o, h, l, c, v }
input.candles["15m"]  // t = open time UTC ms
input.candles["4h"]

// input.referenceLevels
input.referenceLevels.pdh
input.referenceLevels.pdl
input.referenceLevels.sessionOpen

// input.params — merged from analyzer defaults + ruleset overrides
input.params

// input.helpers (functions — call these, do not redefine)
input.helpers.getCandlePosition(/* ... */)
input.helpers.findFirstInteraction(/* ... */)
input.helpers.getCandlesInWindow(/* ... */)
input.helpers.hasWickTouch(/* ... */)
```

## Rules

- **No imports, no `require`, no filesystem or network.** Only pure logic + `input`.
- **No infinite loops.** Execution times out after a few seconds.
- Prefer **defensive checks**; throw clear `Error` messages if invariants fail.
- Keep `label` stable across versions when semantics are unchanged (helps compare rulesets).

## Minimal example

```javascript
function analyze(input) {
  var n = input.candles["15m"].length;
  return {
    label: n > 0 ? "HAS_DATA" : "NO_DATA",
    details: { barCount: n },
  };
}
```

## Native S2 reference

The built-in **native_s2** analyzer runs TypeScript `BehaviorAnalyzer` on the server; it is not generated by an LLM. Use it as a reference for domain vocabulary (sessions, PDH/PDL, decision labels) when specifying new sandbox rules in natural language.
